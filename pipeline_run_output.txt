### Conceptual Guide for HW3: Multi-threaded Web Server and Synchronization

#### Overview
In this assignment, you will develop a multi-threaded web server that efficiently handles multiple HTTP requests concurrently while maintaining a log of handled jobs. The core concepts revolve around multi-threading, synchronization, and effective resource management. Understanding how these concepts interconnect will help you design a robust solution.

#### Key Concepts and Their Connections

1. **Multi-threading and Thread Pool**: 
   - The server must handle multiple requests simultaneously. Instead of creating a new thread for each request (which is inefficient), you will implement a **fixed-size thread pool**. This allows a set number of worker threads to process requests, enhancing performance and resource management.

2. **Producer-Consumer Model**: 
   - The architecture follows a **producer-consumer model** where the master thread (producer) accepts incoming requests and places them into a bounded FIFO queue, while worker threads (consumers) process these requests. This model helps in decoupling request acceptance from processing, allowing for better scalability.

3. **Synchronization**: 
   - With multiple threads accessing shared resources (like the request queue and the server log), **synchronization** is crucial to prevent race conditions. You will use:
     - **Mutexes** to ensure exclusive access to shared resources.
     - **Condition variables** to manage thread waiting and signaling, preventing busy-waiting.

4. **Reader-Writer Lock**: 
   - Since the server log will be accessed by both GET and POST requests, implementing a **reader-writer lock** is essential. GET requests (writers) require exclusive access to append entries, while POST requests (readers) can read concurrently. You will prioritize writers to prevent starvation of log updates.

5. **Error Checking and Logging**: 
   - Robust error checking is critical in a multi-threaded environment to ensure that unexpected issues do not lead to crashes. Additionally, logging request statistics is important for monitoring server performance and debugging. You will implement structured logging to capture request handling times and other metrics.

6. **UDP Communication**: 
   - Beyond handling HTTP requests, your server will also support UDP communication for statistics collection. This involves managing separate threads for handling UDP pings, which should be prioritized to respond quickly to administrative requests without blocking HTTP processing.

7. **Debugging Sleep**: 
   - To simulate real-world conditions, you will implement a debugging feature that introduces a controlled sleep during log read/write operations. This helps in testing the server's performance under load and ensures that your synchronization mechanisms are functioning correctly.

#### Why These Concepts Matter
Understanding and effectively implementing these concepts will lead to a well-structured, efficient, and scalable web server. The multi-threading model allows for handling multiple requests simultaneously, improving user experience. Synchronization ensures data integrity and prevents race conditions, which are common pitfalls in concurrent programming. The logging and statistics collection will provide insights into server performance, allowing for future optimizations.

### Implementation Strategy
- **Start by setting up the thread pool** and implementing the producer-consumer model with a bounded queue.
- **Focus on synchronization mechanisms** (mutexes and condition variables) to manage access to shared resources.
- **Implement the reader-writer lock** for the server log, ensuring that GET and POST requests are handled appropriately.
- **Integrate error checking and logging** throughout your code to maintain robustness.

    queue.front = (queue.front + 1) % QUEUE_SIZE;
    queue.count--;
    pthread_cond_signal(&queue.not_full);
    pthread_mutex_unlock(&queue.mutex);

#include <stdlib.h>

pthread_cond_t log_cond;

    while (writers_count > 0) {
        pthread_cond_wait(&log_cond, &log_mutex);
    }
    readers_count++;
    pthread_mutex_unlock(&log_mutex);

    }
    pthread_mutex_unlock(&log_mutex);

    server_log.readers--;

        return EXIT_FAILURE;

    char buffer[1024];

    // Create socket

        return EXIT_FAILURE;
    }

    server_addr.sin_family = AF_INET;
    server_addr.sin_port = htons(port);
    server_addr.sin_addr.s_addr = INADDR_ANY;

    // Send message to the server
    sendto(sockfd, message, strlen(message), 0, (const struct sockaddr *)&server_addr, sizeof(server_addr));

    // Receive echoed message from the server
    socklen_t len = sizeof(server_addr);
    int n = recvfrom(sockfd, buffer, sizeof(buffer) - 1, 0, (struct sockaddr *)&server_addr, &len);
    buffer[n] = '\0'; // Null-terminate the received string

    // Assert that the echoed message is correct
    if (strcmp(buffer, message) == 0) {
        printf("Test passed: Received expected echo: %s\n", buffer);
    } else {
        printf("Test failed: Expected '%s', but received '%s'\n", message, buffer);
    }

    // Clean up
    close(sockfd);
    pthread_cancel(server_thread);
    pthread_join(server_thread, NULL);

    return EXIT_SUCCESS;
